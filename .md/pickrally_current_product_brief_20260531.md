# PickRally 현재 제품 브리프

작성일: 2026-05-31

## 목적

이 문서는 현재 저장소의 실제 구현 상태와 제출용 기획 방향을 한 곳에 정리한다. 기존 `rallyroom_*` 파일명은 초기 코드네임 이력을 유지한 것이며, 현재 서비스명은 `PickRally`이다.

## 한 줄 설명

PickRally는 팬이 직접 투표방을 열고, 인기투표·미션·팬월·결과 카드로 참여를 반복하는 팬 주도 투표 커뮤니티 서비스다.

## 문제 정의

팬 커뮤니티는 재방문을 만들려면 단순 소개 페이지보다 반복 참여 루프가 필요하다. 실존 스타나 작품과 공식 제휴가 없는 과제형 MVP에서는 공식 투표처럼 보이는 구조를 피해야 한다. PickRally는 “공식 영향력”이 아니라 “팬이 직접 만든 인기투표와 기록 카드”를 중심으로 사용자의 참여 동기를 만든다.

## 제품 구조

- 투표방: 팬이 만든 지속 공간
- 현재 투표: 투표방 안의 활성 세션
- 완료 투표: 결과/기록 탭과 결과 카드로 축적되는 히스토리
- 팬월: 참여 메시지와 보상 루프
- 미션: 재방문과 행동 유도
- RP/투표권: 참여 보상과 추가 참여를 연결하는 내부 재화
- Crew dashboard: 운영자/파트너 확장 가능성을 보여주는 aggregate view

MVP에서는 한 투표방에 활성 투표 1개만 노출한다. 이후 같은 형식의 투표를 프리셋으로 재사용하거나, 완료 투표 기록을 누적하는 방향으로 확장한다.

## 현재 구현 범위

현재 `main` 기준으로 프론트엔드 목업은 다음을 포함한다.

- 홈: Featured 투표방, 카테고리 필터, 투표방 카드 탐색
- 투표방 만들기: 초기 후보 무료 추가, Enter 후보 추가, IME/Safari 조합 입력 guard, 서버 요청 미리보기와 접수 예시
- 방 상세: 현재 투표, 팬월, 미션, 결과/기록 탭
- 투표: 세로 후보 리스트, 투표권 선택, 반복 투표, Vote Energy 증가와 마감 처리
- 후보 추가: 활성 투표 중 투표권을 사용한 inline 후보 추가와 자동 투표
- RP 교환: 100 RP를 투표권 1장으로 교환하는 목업 플로우
- 팬월: 메시지 작성 경계와 보상 응답 예시
- 미션: 미션 완료 경계와 보상 응답 예시
- 결과 카드: 공개 결과 카드와 방장 발행 요청 플로우
- 프로필: 최근 보상, 참여 중인 투표, 완료된 투표 카드, 프로필 수정 목업과 로그인 guard
- 요금제: 구독/투표권 패키지/Crew 문의 요청 preview
- Crew: raw event가 아닌 aggregate dashboard

## 비즈니스 방향

단기 제출에서는 실제 결제보다 “과금 가능한 반복 루프”를 보여주는 것이 핵심이다.

- 무료 사용자: 제한된 투표권, 공개 투표방 참여, 기본 결과 카드
- Plus 사용자: 추가 투표권, RP 보너스, 결과 카드 테마, 빠른 후보 추가
- Crew/파트너: 투표방 운영 도구, aggregate dashboard, 공식 계정/브랜드 페이지, moderation queue
- 향후 공식 계정: 팬 주도 투표방과 공식 운영 투표방을 구분해 공식성 오인을 줄임

## 현실적 제약

- 실존 스타, 작품, 방송 이미지, 공식 로고를 사용하지 않는다.
- 투표 결과가 외부 스타/작품에 전달된다는 표현을 쓰지 않는다.
- 현재 구현은 demo read model 기반이며, 실제 Supabase DB write는 아직 연결하지 않는다.
- 투표수, RP, Vote Energy 같은 신뢰 데이터는 프론트에서 최종 권위로 계산하지 않는다.

## 현재 정리된 사소한 구멍

- 서비스명은 UI, HTML title, 문서, `.env.example` 모두 `PickRally`를 기준으로 맞춘다.
- `RallyRoom`은 초기 코드네임이므로 파일명, 타입명, 역사적 참고 문서에만 남긴다.
- 관심 카테고리 수정은 별도 `/profile/categories`가 아니라 `/profile/edit` 목업 안에서 처리한다.

## 제출 포지셔닝

제출 시에는 “완전한 운영 서비스”가 아니라, 팬 주도 투표 커뮤니티의 핵심 루프와 화면 구조를 VibeX로 빠르게 검증한 MVP라고 설명한다. 평가 포인트는 구현 완성도보다 문제 구조화, 반복 참여 루프, AI 협업 과정, 후속 백엔드 확장 가능성에 둔다.
