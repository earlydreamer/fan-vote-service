# PickRally 제출용 주요 프롬프트와 smoke 체크

작성일: 2026-05-30
최종 갱신일: 2026-05-31

> 파일명은 초기 코드네임 `RallyRoom` 이력을 유지한다. 현재 서비스명과 UI 표기는 `PickRally`를 우선한다.

## 목적

VibeX 과제 제출 항목 중 “사용한 주요 프롬프트”와 제출 직전 smoke 체크를 정리한다. 제출 메일에는 이 문서의 요약본을 붙이면 된다.

## 제출용 서비스 설명

PickRally는 팬이 직접 투표방을 만들고, 인기투표·미션·팬월·결과 카드로 참여 기록을 쌓는 팬 주도 투표 커뮤니티 서비스다. MVP는 실존 스타/작품과 공식 제휴를 전제로 하지 않고, 가상의 주제와 데모 데이터를 사용한다. 핵심은 공식 영향력이 아니라 팬이 직접 만든 투표 콘텐츠, 반복 참여, 보상 루프, 결과 공유다.

## 핵심 화면 흐름

1. 홈에서 Featured 투표방, 카테고리 필터, 투표방 카드 그리드, 마감 임박 투표를 탐색한다.
2. 투표방 만들기에서 방 이름, 현재 투표 제목, 카테고리, 후보, 마감, 보상 조건을 입력하고 `create-room` 요청 preview와 접수 예시를 확인한다.
3. 방 상세에서 `현재 투표`, `팬월`, `미션`, `결과/기록` 탭을 전환한다.
4. 현재 투표에서 투표권을 선택해 반복 투표하고, Vote Energy가 목표치에 도달하면 투표가 마감되는 흐름을 확인한다.
5. 활성 투표 중 후보를 추가하면 투표권을 사용하고, 사용한 투표권만큼 새 후보에 자동 투표된다.
6. RP를 투표권으로 교환해 보상과 참여가 연결되는 루프를 확인한다.
7. 팬월 메시지와 미션 완료는 서버 처리 경계와 보상 응답 예시로 동작한다.
8. 결과 카드 화면에서 공개 결과와 방장 발행 요청 흐름을 확인한다.
9. 프로필에서 최근 보상, 참여 중인 투표, 완료된 결과 카드, 프로필 수정 진입점을 확인한다.
10. Crew 화면에서 raw event가 아닌 aggregate dashboard를 확인한다.
11. 요금제 화면에서 Plus, 투표권 패키지, Crew 문의 요청 preview를 확인한다.

## 사용한 주요 프롬프트 요약

### 1. 기획 정렬

```text
팬 커뮤니티 주제 중 비즈니스와 연결될 수 있는 형태를 찾고 싶다.
단순 랜딩 페이지나 정보성 페이지가 아니라 정기 재방문과 과금 가치가 있는 구조여야 한다.
실존 스타와 제휴하지 않은 상태에서 공식성 오인을 만들지 않으면서 팬이 직접 투표방을 만들고 참여하는 구조를 비판적으로 검토해 달라.
투표방 단위, 투표, 리워드, 공식 계정 확장 가능성, 현실적 제약과 보완 방향을 함께 정리해 달라.
```

### 2. 제품 구조 재정렬

```text
응원이라는 키워드에 매몰되지 말고 실제 행위는 Vote에 가깝게 보이도록 정리해 달라.
방은 지속되는 팬 공간, 투표는 방 안의 현재 세션으로 분리한다.
MVP에서는 한 방에 활성 투표 1개만 보여주되 완료 투표 기록과 프리셋 확장이 가능한 UX로 표현한다.
방 정보, 현재 투표, 참여/항목 추가, 팬월/미션/기록의 위계를 명확히 해 달라.
```

### 3. 디자인 방향 재설정

```text
대시보드처럼 답답한 레이아웃이 아니라 콘텐츠 탐색 플랫폼처럼 시원하게 보이게 해 달라.
CHZZK와 SOOP은 구조 참고만 하고, 다크모드나 실제 이미지/브랜드는 복제하지 않는다.
Featured, 카테고리 탐색, 투표방 카드, 컴팩트한 사용자 상태, 투표방 만들기 진입점을 포함하되 고정 2단 레이아웃을 강제하지 않는다.
목데이터 수량을 늘리고 후보 수, 투표 형식, 카테고리를 다양하게 구성한다.
```

### 4. 프론트엔드 구현

```text
React/Vite/TypeScript 기반으로 VibeX 제출용 MVP를 구현한다.
홈, 투표방 만들기, 방 상세, 투표, 후보 추가, 미션, 팬월, 결과 카드, 프로필, Crew 대시보드, 요금제 화면을 demo data 기반으로 연결한다.
프론트엔드는 vote_count, reward_rp, total_rp, current_goal_value 같은 신뢰 필드를 직접 쓰지 않고 서버 명령 API 경계를 보여준다.
모든 구현은 TDD 기반으로 진행하고, React Best Practice 기준으로 검증한다.
```

### 5. 투표 UX 개선

```text
투표 후보는 그리드가 아니라 세로 리스트로 보여준다.
투표권을 1장 이상 선택해 투표하고, 사용한 투표권만큼 후보 표와 Vote Energy가 증가한다.
Vote Energy가 목표치에 도달하면 투표가 마감된다.
방 생성 시 초기 후보 추가는 무료이고, 이미 시작한 투표에 후보를 중간 추가할 때만 투표권을 사용한다.
중간 추가한 후보에는 사용한 투표권만큼 자동 투표가 들어가야 한다.
```

### 6. 제출 준비

```text
VibeX 제출 전에 주요 프롬프트, 화면 흐름, 완성 범위, 검증 명령, 보류 범위를 정리한다.
현재 MVP는 demo data와 서버 명령 API 경계 중심이며, 실제 Supabase schema/RLS/RPC/Edge Function 배포는 후속 백엔드 feature로 둔다.
```

## 로컬 smoke 체크리스트

로컬 서버:

```bash
npm run dev -- --host 127.0.0.1 --port 5173
```

체크 대상:

- `/`: 홈 탐색, Featured, 카테고리 필터, 투표방 카드
- `/rooms/new`: 투표방 만들기, 후보 추가, IME/Enter 처리, 서버 요청 접수 예시
- `/rooms/room-stage-opening`: 현재 투표, 반복 투표, 후보 추가, 팬월, 미션, 결과/기록 탭
- `/rooms/room-pixel-season/result`: 공개 결과 카드
- `/rooms/room-closed-finale/result`: 결과 카드 발행 요청
- `/login`: 회원가입/로그인 목업과 데모 계정 CTA
- `/profile`: 보상 이력, 참여 중인 투표, 완료된 결과 카드, 프로필/관심 카테고리 수정 진입점
- `/profile/edit`: 로그인 guard와 닉네임/관심 카테고리 수정 목업
- `/crew`: aggregate 운영 대시보드
- `/pricing`: 요금제, RP/투표권 교환과 구매/문의 요청 preview
